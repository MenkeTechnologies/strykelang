# RFC-0001: Package Registry

- **Status**: Draft
- **Created**: 2026-04-26
- **Author**: MenkeTechnologies

## Summary

A pnpm-style package registry for stryke with content-addressable store, symlinked vendor directories, and no duplication of transitive dependencies.

## Motivation

stryke needs to evolve from a language into an ecosystem. A package registry is the foundation for:

- Code sharing and reuse
- Dependency management
- Reproducible builds
- Community growth

npm's flat `node_modules` with hoisting is a disaster (phantom deps, version conflicts, massive disk usage). We adopt pnpm's proven model from day one.

## Design

### Core Principles

1. **Content-addressable store** — one copy per package version globally
2. **Symlinked deps** — projects link to store, no duplication
3. **Deterministic resolution** — lock file ensures reproducible installs
4. **Offline-capable** — works from cache without network
5. **Local-first** — full functionality without remote registry

### Directory Layout

#### Global Store (`~/.stryke/`)

```
~/.stryke/
├── store/                      # extracted packages (content-addressable)
│   ├── http@1.0.0/
│   │   ├── lib/
│   │   └── stryke.toml
│   ├── json@2.1.0/
│   └── crypto@0.5.0/
├── bin/                        # global CLI tools
│   └── mytool -> ../store/mytool@1.0.0/bin/mytool
├── cache/                      # downloaded tarballs
│   └── http-1.0.0.tar.zst
└── git/                        # cloned git deps
    └── github.com/
        └── user/
            └── repo/
```

#### Project Layout

```
myproject/
├── stryke.toml                 # manifest (checked in)
├── stryke.lock                 # lock file (checked in)
├── lib/                        # project source
│   └── main.stk
└── vendor/                     # symlinks to store (gitignored)
    ├── http -> ~/.stryke/store/http@1.0.0
    └── json -> ~/.stryke/store/json@2.1.0
```

### Manifest Format (`stryke.toml`)

```toml
[package]
name = "myapp"
version = "0.1.0"
description = "My stryke application"
authors = ["user@example.com"]
license = "MIT"
repository = "https://github.com/user/myapp"
stryke = ">=0.9.0"              # minimum stryke version

[deps]
http = "1.0"                    # semver: ^1.0.0
json = "=2.1.0"                 # exact version
crypto = { version = "0.5", optional = true }

[deps.local-lib]
path = "../my-local-lib"

[deps.git-lib]
git = "https://github.com/user/lib"
tag = "v1.0.0"

[dev-deps]
test-utils = "1.0"

[build-deps]
codegen = "0.2"

[scripts]
test = "s test t/"
build = "s build lib/main.stk -o dist/myapp"
lint = "s check lib/"
fmt = "s fmt -i lib/"

[bin]
myapp = "lib/main.stk"

[features]
default = ["json"]
full = ["json", "crypto"]
```

### Lock File Format (`stryke.lock`)

```toml
# Auto-generated by stryke. Do not edit.
version = 1
stryke_version = "0.9.3"

[[package]]
name = "http"
version = "1.0.0"
source = "registry"
integrity = "sha256-a3f2b8c9d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2"
deps = [
    "json@2.1.0",
]

[[package]]
name = "json"
version = "2.1.0"
source = "registry"
integrity = "sha256-b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6"
deps = []

[[package]]
name = "crypto"
version = "0.5.0"
source = "registry"
integrity = "sha256-c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7"
deps = [
    "json@2.1.0",
]

[[package]]
name = "local-lib"
version = "0.0.0"
source = "path:../my-local-lib"
deps = []

[[package]]
name = "git-lib"
version = "1.0.0"
source = "git:https://github.com/user/lib?tag=v1.0.0"
rev = "abc123def456"
deps = []
```

### CLI Commands

```bash
# Install all deps from stryke.toml
s install
s install --offline           # use only cached packages
s install --frozen            # fail if lock file would change

# Add dependency
s add http                    # latest version
s add http@1.0.0              # specific version
s add http --dev              # dev dependency
s add http --optional         # optional feature dep
s add ../local-lib --path     # local path dep
s add https://github.com/u/r --git  # git dep

# Remove dependency
s remove http

# Update dependencies
s update                      # all, respecting semver
s update http                 # specific package
s update --latest             # ignore semver, get latest

# Global tools
s install -g mytool
s uninstall -g mytool
s list -g                     # list global tools

# Info
s deps                        # list direct deps
s deps --tree                 # dependency tree
s deps --why json             # why is json installed?
s info http                   # package metadata

# Scripts
s run test                    # run script from [scripts]
s run build

# Publishing (future)
s publish
s publish --dry-run

# Maintenance
s prune                       # remove unused from store
s cache clean                 # clear download cache
s audit                       # security check (future)
```

### Resolution Algorithm

1. **Parse** — read `stryke.toml`, collect direct deps
2. **Expand** — recursively fetch transitive deps' manifests
3. **Resolve** — apply semver constraints:
   - Prefer highest compatible version
   - Deduplicate: `^2.0` + `^2.1` → `2.1.x`
   - Conflict detection: `1.x` + `2.x` → error
4. **Lock** — record exact versions in `stryke.lock`
5. **Fetch** — download missing tarballs to cache
6. **Verify** — check integrity hashes
7. **Extract** — unpack to store
8. **Link** — create `vendor/` symlinks

### Module Resolution

When stryke encounters `use Foo::Bar`:

1. `lib/Foo/Bar.stk` — project source
2. `vendor/foo/lib/Bar.stk` — dependency
3. `@INC` paths — system/site libs

The resolver maps package names to vendor paths automatically.

### Deduplication

```
# Same json version used by http and crypto
~/.stryke/store/json@2.1.0/     # single copy

project-a/vendor/http/          # symlink
project-a/vendor/crypto/        # symlink
project-b/vendor/json/          # same physical files

# Disk usage: O(unique versions), not O(total deps)
```

### Integrity & Security

- SHA-256 hash of compressed tarball
- Verified on every install
- Lock file records hashes for reproducibility
- Tampered packages fail immediately

### Git Dependencies

```toml
[deps.mylib]
git = "https://github.com/user/mylib"
branch = "main"                 # default
tag = "v1.0.0"                  # or specific tag
rev = "abc123"                  # or exact commit
```

Cloned to `~/.stryke/git/{host}/{user}/{repo}/`, checked out to requested ref, linked to store.

### Path Dependencies

```toml
[deps.mylib]
path = "../mylib"
```

Direct symlink to local path. Changes reflect immediately. Not published to registry.

### Workspaces (Future)

```toml
[workspace]
members = [
    "packages/*",
    "apps/*",
]

[workspace.deps]
http = "1.0"                    # shared version
```

Monorepo support with:
- Single `stryke.lock` at root
- Shared dep versions
- Cross-package local refs

### Registry Protocol (Future)

```
GET  /api/v1/packages/{name}
GET  /api/v1/packages/{name}/{version}
GET  /api/v1/packages/{name}/{version}/download
POST /api/v1/packages (auth required)
```

Simple HTTP API. Package format: zstd-compressed tarball.

## Alternatives Considered

### npm-style flat node_modules
Rejected. Hoisting causes phantom dependencies, version conflicts, nondeterministic installs.

### Cargo-style compilation
Not applicable. Cargo compiles deps into binary; stryke interprets/JITs at runtime. Need runtime linking.

### Go modules
Considered. GOPATH → modules transition was painful. Starting fresh with pnpm model is cleaner.

### Deno-style URL imports
Considered. Simple but no versioning, no offline, no deduplication. Registry is better long-term.

## Open Questions

1. **Package naming** — flat namespace or scoped (`@user/pkg`)?
2. **Binary distribution** — allow precompiled native extensions?
3. **Mirroring** — corporate mirror support from day one?
4. **Signing** — package signatures beyond integrity hash?
